# ADR-001: Padrão de Persistência de Dados (formato Setup Orion)

**Status**: Aceito
**Data**: 2024 (revisado 2026-06-18 — alinhado ao Setup Orion; supersede o formato Markdown `.md`)

---

## Contexto

O ecossistema Setup Orion (consolidado, ver `docs/SetupOrion.md`) persiste os dados de
instalação em **arquivos de texto plano sem extensão**, nomeados `dados_<servico>`, em
`/root/dados_vps/`, no formato `[ CABEÇALHO ]` + pares `Campo: valor`. As skills dependentes
leem credenciais e parâmetros via `grep "Campo:" | awk`.

A revisão anterior deste ADR adotava Markdown (`<skill>.md`), que **divergiu do padrão
consolidado** e quebrou a interoperabilidade (leituras `grep "Senha:"` não casavam com
`- **Senha**:`, nomes com prefixo `app-`/`infra-` não batiam). Decidimos **seguir o padrão
Setup Orion** para manter compatibilidade e previsibilidade.

---

## Decisão

Toda persistência usa **`/root/dados_vps/dados_<servico>`** (sem extensão), onde `<servico>`
é o nome curto da stack (sem os prefixos `app-`/`infra-`).

### Convenção de caminho

```
/root/dados_vps/dados_<servico>
```

Exemplos:
- `/root/dados_vps/dados_traefik`
- `/root/dados_vps/dados_portainer`
- `/root/dados_vps/dados_postgres`
- `/root/dados_vps/dados_mongodb`

Há ainda o arquivo global `/root/dados_vps/dados_vps` (nome do servidor, rede interna,
email SSL, link do Portainer), gerado pelo `app-traefik`.

### Formato do conteúdo (Setup Orion)

```
[ MONGODB ]

Dominio do MongoDB: mongodb://root:<senha>@mongodb:27017/?authSource=admin

Host: mongodb

Port: 27017

Usuario: root

Senha: <senha>
```

Leitura padronizada: `grep "Senha:" /root/dados_vps/dados_mongodb | awk -F"Senha:" '{print $2}' | xargs`.

### Camada de abstração: `lib-persistence.sh`

Toda deploy skill **deve** carregar a biblioteca antes de qualquer operação:

```bash
SKILL_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
source "$SKILL_DIR/../00-core/lib-persistence.sh"
```

| Função | Uso |
|--------|-----|
| `save_data "<servico>" "<conteúdo>"` | Escreve/sobrescreve `dados_<servico>` (chmod 600) |
| `read_data "<servico>"` | Lê o arquivo `dados_<servico>` |
| `service_exists "<servico>"` | Verifica se `dados_<servico>` existe |
| `deploy_via_portainer "<stack>" "<yaml>"` | Deploy via API do Portainer (ver ADR-003) |

O nome passado às funções pode incluir prefixo (`infra-mongodb`); a lib normaliza para
`dados_mongodb` automaticamente.

### `metadata.json`

Cada skill declara o caminho no novo padrão:

```json
{
  "persistence_path": "/root/dados_vps/dados_minhaskill"
}
```

---

## Consequências

**Prós**:
- Compatível com o Setup Orion consolidado (leitura `grep`/`awk` entre skills funciona)
- Credenciais de dependências (Postgres/Mongo/etc.) acessíveis a stacks dependentes
- Auditável em shell: `cat /root/dados_vps/dados_*`

**Contras**:
- Arquivos passam a conter segredos → ver **ADR-002** (revisado): `chmod 600` + `/root` restrito
- Não é transacional — sobrescrita concorrente pode corromper

---

## Alternativas Rejeitadas

- **Markdown `.md`** (revisão anterior): divergiu do padrão Setup Orion e quebrou interoperabilidade
- **SQLite**: overhead desnecessário para o volume de dados esperado
- **Variáveis de ambiente persistidas**: não sobrevivem à reinicialização do host
