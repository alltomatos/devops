# ADR-002: Segurança de Segredos (persistência operacional no padrão Setup Orion)

**Status**: Aceito
**Data**: 2024 (revisado 2026-06-18 — alinhado ao Setup Orion; revoga a proibição total de persistir segredos)

---

## Contexto

O projeto lida com credenciais de banco de dados, tokens de API, senhas SMTP e chaves de
aplicação. A revisão anterior proibia **persistir qualquer segredo** (contexto efêmero).

Na prática, o ecossistema Setup Orion (consolidado) **persiste** as credenciais em
`/root/dados_vps/dados_<servico>` — é assim que stacks dependentes obtêm a senha do
Postgres/Mongo/etc. sem re-perguntar a cada deploy. A proibição total tornava o modelo de
dependências inoperante (apps liam `grep "Senha:"` de arquivos que nunca continham a senha).

Decidimos **seguir o padrão Setup Orion**: segredos operacionais ficam em disco em
`/root/dados_vps/`, protegidos por permissão de arquivo e pelo isolamento de `/root`.

---

## Decisão

### 1. Proibição de hardcode no repositório (mantida)

Nenhum segredo pode aparecer fixo em `run.sh`, `metadata.json`, `.md` do repo ou commits.
Segredos são **gerados em runtime** (`openssl rand`) ou recebidos via env do chamador.
**O repositório git nunca contém segredos.**

### 2. Persistência operacional na VPS (NOVO)

Segredos de instalação **são persistidos** em `/root/dados_vps/dados_<servico>`:
- Formato Setup Orion (`Senha: <valor>`, `Token: <valor>`) — ver ADR-001.
- Arquivos com **`chmod 600`** (gravados por `save_data`/`portainer_save_auth`).
- Proteção em camadas: `/root` é acessível apenas por root; arquivos não são world-readable.

| Dado | Persistir em `dados_<servico>`? |
|------|--------------------------------|
| URL de acesso, usuário, host, porta, rede | ✅ Sim |
| Senha / token / chave de app | ✅ Sim (chmod 600) |
| Segredo no repositório git | ❌ Nunca |

### 3. Fluxo de um segredo

```
gerado em runtime (openssl) ou recebido via env
      ↓
embutido no YAML e enviado ao Portainer/Swarm (ADR-003)
      ↓
persistido em /root/dados_vps/dados_<servico> (chmod 600)
      ↓
lido por skills dependentes via grep/awk
```

### 4. Mascaramento em logs de chat (mantido)

Ao receber um segredo do usuário, o agente confirma sem repetir o valor
(`"Senha recebida ✓"`) e nunca o exibe em resumos. Ao exibir arquivos `dados_*`,
mascarar `Senha:`/`Token:`.

### 5. Higiene

- `dados_portainer` e demais `dados_*` com segredo: sempre `chmod 600`.
- Senhas efêmeras de encadeamento (ex.: passar a senha do Mongo a um app no mesmo deploy)
  podem usar arquivo temporário `chmod 600` removido logo após (`shred -u`).

---

## Consequências

**Prós**:
- Compatível com o Setup Orion; modelo de dependências entre stacks funciona
- Re-deploy não exige re-fornecer toda credencial (lê de `dados_<servico>`)
- Segredos seguem fora do git (gerados em runtime)

**Contras**:
- Segredos em repouso na VPS (mitigado por `chmod 600` + `/root` restrito + backup/criptografia de disco recomendados)
- Quem tiver root na VPS lê os segredos (inerente a qualquer orquestrador no host)

---

## Alternativas Rejeitadas

- **Proibição total de persistência** (revisão anterior): quebrou o modelo de dependências do Setup Orion
- **Docker Swarm secrets**: tecnicamente viável, mas diverge do padrão Setup Orion consolidado
- **`.env` no repo / segredos commitados**: risco direto de vazamento via git
